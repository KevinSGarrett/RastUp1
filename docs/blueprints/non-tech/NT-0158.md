G\) Producer Obligations

Emit at least the events listed above for the flows you implement.

Include the envelope fields, correct privacy_tier, and validated payload
per schema.

Idempotency: do not reemit the same event_id; retries must reuse the
event_id.

Ordering: preserve perentity ordering (e.g., for a given booking_leg_id)
when feasible.

H\) Schema Registry & CI

JSON Schemas (per event_name & event_version) live in the Schema
Registry; publishers fetch at build.

CI checks:

Validate all emitted test payloads against registry.

Block PRs that remove required fields or introduce forbidden PII.

Require a diff note describing metric impact for any event change (ties
to ยง5.6 catalog).

I\) Backward Compatibility & Deprecation

Add fields as optional first; only make required after a tworelease
grace period and consumer readiness.

For breaking semantics, mint a new event_name (e.g., \_v2) and run both
for one release cycle, then deprecate older.

The Registry marks deprecated with an EOL date; ingestion continues but
dashboards switch based on the catalog version.

J\) Acceptance Criteria (mark ยง5.1 FINAL)

Envelope and required fields implemented by all producers (web, mobile,
services, admin).

Naming & semver rules enforced; namespaces aligned with product modules.

Privacy tiers validated; ingestion blocks forbidden PII.

JSON Schemas for the 12 core families published to the Registry with
examples.

CI gating for schema validation & "no breaking changes without new
name."

Sample payloads validated endtoend into Bronze; test tables show
expected rows.

K\) Why this works

Stable for consumers (dashboards, finance, ML) because producers evolve
additively.

Auditable because every event has consistent IDs, timestamps, and
privacy classification.
